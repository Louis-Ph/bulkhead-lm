# GitHub Repository Settings

This document captures the recommended GitHub-side configuration for the public
`bulkhead-lm` repository so the UI settings, merge policy, and repo files stay
aligned.

## General

- repository name: `bulkhead-lm`
- default branch: `main`
- template repository: off
- release immutability: on
- social preview: on, with a custom `1280x640` image
- preserve this repository: on

Suggested repository description:

`Security-first OCaml LLM gateway with explicit provider routing, virtual keys, fail-closed egress, and OpenAI-compatible endpoints.`

Suggested topics:

- `ocaml`
- `llm`
- `llm-gateway`
- `openai-compatible`
- `security`
- `cli`
- `ssh`
- `routing`

## Features

- wikis: off
- issues: on
- discussions: on
- projects: off until active multi-issue planning justifies it
- pull requests: on
- sponsorships: off unless a real funding target is published

## Pull Requests

- allow merge commits: off
- allow squash merging: on
- allow rebase merging: off
- always suggest updating pull request branches: on
- allow auto-merge: on
- automatically delete head branches: on

Recommended squash commit message style:

- default to pull request title

## Commits

- require signoff on web-based commits: off unless a repository-wide DCO policy is introduced
- allow comments on individual commits: on

## Issues

- auto-close linked issues with merged pull requests: on

## Archives

- include Git LFS objects in archives: off

## Branch Protection Or Ruleset

Apply a ruleset to `main` with these requirements:

- block force pushes
- block branch deletion
- require pull requests before merge
- require at least 1 approval
- require review dismissal on new commits
- require conversation resolution before merge
- require status checks to pass before merge
- require the `Build And Test (Ubuntu)` workflow check
- require linear history if squash-only policy is enforced
- restrict direct pushes to maintainers only, or disable direct pushes entirely

## Security

- enable private vulnerability reporting
- keep `SECURITY.md` as the public reporting policy
- consider enabling Dependabot alerts and Dependabot security updates

## Discussions

If Discussions is enabled, use it for:

- how-to questions
- beginner setup help
- architecture discussion before a large feature request

Keep bugs and security-sensitive material out of Discussions.

## Releases

- use immutable releases
- use autogenerated release notes
- keep `.github/release.yml` aligned with the label taxonomy
